# Orz Markdown HTML Files

Use this skill for the file model and editing rules of `.md.html` documents.

Do not use it for Orz Markdown syntax itself. If the task is about plugin syntax, markdown containers, KaTeX notation, Mermaid authoring, or other parser features, load the separate Orz Markdown syntax skill as well.

## Core model

Treat a `.md.html` file as a dual-format document:

- the real file on disk is HTML,
- the editable source of truth is markdown,
- that markdown is stored inside `<script type="text/markdown" id="md-source">...</script>`,
- the rendered `<article class="markdown-body">` content is generated output.

This distinction matters because the extension regenerates the rendered HTML whenever the markdown changes. Editing the generated article directly is therefore fragile unless the user explicitly wants raw-HTML surgery.

## Default workflow

When the user asks to create or edit a `.md.html` file, follow this order:

1. Prefer editing the markdown source block instead of the rendered HTML.
2. Preserve the surrounding HTML shell unless the task is to repair or rebuild it.
3. Preserve theme metadata and runtime assets so the file stays compatible with the extension.
4. Escape any literal `</script>` inside markdown as `<\/script>`.

## What to edit

For an existing, extension-managed file, change only the markdown payload inside:

```html
<script type="text/markdown" id="md-source">
...markdown source...
</script>
```

Keep these parts intact unless the task is specifically about repairing the shell:

- `<meta name="orz-md-theme-index" content="N">`
- `<html ... data-ui-scheme="dark|light">`
- the inline theme `<style id="theme-*">`
- `<article class="markdown-body">...</article>`
- the runtime script tags
- `<link rel="stylesheet" href="custom.css">`

Why: those pieces are either regenerated by the extension or used to restore theme and output behavior when the file is reopened.

## Creating new files

Prefer the extension workflow when possible:

1. Create an empty `.md.html` file.
2. Open it with the Orz Markdown extension.
3. Let the extension auto-initialize the HTML shell.
4. Edit the markdown view.

If manual bootstrapping is required, create a valid HTML document that includes:

- `data-ui-scheme` on `<html>`
- a `<meta name="orz-md-theme-index" content="0">` tag or another valid theme index
- inline theme CSS in a `<style id="theme-*">` block
- a rendered `<article class="markdown-body">`
- the markdown source block with id `md-source`
- the usual runtime library/script hooks
- `<link rel="stylesheet" href="custom.css">`

Use theme index `0` as a safe default when no theme is specified.

## Repairing broken files

If a `.md.html` file is empty or missing the markdown source block:

- repair it by restoring a valid shell and an empty markdown source block,
- do not invent user content,
- keep recoverable metadata when present.

If only the rendered article looks stale but the markdown block is intact, update the markdown block and let the extension regenerate the rest.

## Theme mapping

Use these indices when you need to reason about saved theme metadata:

| Index | Theme | Scheme |
|-------|-------|--------|
| 0 | Dark Elegant I | dark |
| 1 | Dark Elegant II | dark |
| 2 | Light Neat I | light |
| 3 | Light Neat II | light |
| 4 | Beige Decent I | light |
| 5 | Beige Decent II | light |
| 6 | Light Academic I | light |
| 7 | Light Academic II | light |
| 8 | Light Playful I | light |
| 9 | Light Playful II | light |

Keep `data-ui-scheme` aligned with the saved theme: dark for `0` and `1`, light for `2` through `9`.

## Preview versus output

Keep these behavior differences in mind when answering users:

- The VS Code preview uses bundled local vendor assets and is intended to work offline.
- The saved `.md.html` output links KaTeX, Mermaid, highlight.js, and SmilesDrawer from CDNs.
- The saved file still opens directly in a browser with no server, but rich rendered features may need network access.
- `custom.css` is loaded by the saved output, not by the VS Code preview.
- Preview font scaling is an editor feature and is not baked into the exported page.

## `custom.css`

If the user wants a folder of `.md.html` files to share a visual override layer, tell them to place `custom.css` in the same directory as those files. The saved page links it as a sibling stylesheet, which makes lightweight static-site style customization possible without changing each document.

## Guardrails

- Do not rewrite the entire HTML shell for a simple content edit.
- Do not delete or rename the `md-source` script block.
- Do not claim the saved output is fully offline for all rich features; CDN-backed libraries are part of the current format.
- Do not describe the rendered article as the source of truth.
- Do not hand-edit generated HTML if the user only asked to change document content.

## Response pattern

When helping with a `.md.html` task, usually do this:

1. Identify whether the task is content editing, shell repair, theme adjustment, or raw HTML inspection.
2. If it is content editing, work inside the markdown source block.
3. If it is shell repair, preserve any recoverable theme metadata and restore the required structure.
4. Mention preview-versus-output differences only when they affect the user’s goal.

## Examples

**Example 1**
Input: “Update this `.md.html` page with a new section and keep the file compatible with the extension.”
Output: Edit only the markdown inside `#md-source`; leave the generated article and shell intact.

**Example 2**
Input: “Create a new `.md.html` handout file I can open in a browser and later edit as markdown.”
Output: Prefer an empty `.md.html` file opened through the extension; if manual creation is required, include the extension-compatible shell and embedded markdown block.

**Example 3**
Input: “Why does my page look different in the browser than in VS Code?”
Output: Explain the preview-versus-output split, including bundled preview assets, CDN-backed browser rendering, and the fact that `custom.css` only applies to saved output.
